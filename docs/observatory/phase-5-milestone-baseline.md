# phase-5-milestone-baseline

Generated by `engine-bench-frame-pacing --run` (see ADR-016, ADR-047).
Captures per-frame wall-clock and rolls up `mean_ms`, `p99_ms`,
`stddev_ms`, `min_ms`, `max_ms` for the Phase-5 milestone scenario.

The Phase 5 milestone is **RX 580 @ 1440p @ 60 FPS** with
`p99 ≤ 18.3 ms` and `σ ≤ 1.04 ms` (ADR-016 + ADR-047 §3). The PR-5
baseline below is the **CPU-oracle** stand-in: it drives the bilinear
upscaler placeholder over a synthetic HDR gradient (1280×720 →
2560×1440) per frame. GPU-backed numbers land when the self-hosted RX
6700 XT runner stands up per ADR-047 §2 — until then the baseline is
informational and the CI gate is `continue-on-error: true` per
ADR-047 §7.

The CPU workload is **not** the same workload the gate measures in
PR 6 onward. CPU numbers will be wildly under-budget (bilinear of
1.8 M pixels is a millisecond-class operation on a Ryzen-class host);
the gate's calibration depends on the full deferred + post-FX +
upscale path running on RDNA-2 hardware. The PR-5 baseline exists to
prove the **measurement methodology** is sound (deterministic
scenario, owned arg parser, owned JSON report, p99/σ stats roll-up,
gate file at `tools/frame-pacing/budgets.toml`) before the runner is
provisioned.

## Bring-up note (2026-05-26, dev machine)

First-light CPU-oracle numbers on the developer machine
(Linux 7.x, CachyOS, BORE scheduler, Ryzen-class). 30-frame scenario
at the milestone resolutions (1280×720 → 2560×1440 bilinear,
single-threaded scalar Rust).

```bash
cargo run --release -p engine-bench-frame-pacing -- --run --frames 30
```

| Field         | Value (dev machine; PR-5 CPU oracle) |
|---------------|--------------------------------------|
| `scenario`    | `phase-5-milestone-cpu-oracle`       |
| `upscaler`    | `owned.bilinear`                     |
| `frames`      | `30`                                 |
| `mean_ms`     | ~ 66 ms (over the 16.6 ms budget)    |
| `p99_ms`      | ~ 95 ms (over the 18.3 ms gate)      |
| `stddev_ms`   | ~ 5.9 ms (over the 1.04 ms gate)     |

The CPU numbers are **over budget by 4-6×** because the scalar
bilinear is upscaling 1.8 M output pixels (≈ 7 MB of `Vec3`)
per frame on a single CPU thread. This is **expected**: the gate
target is RX 580 GPU hardware doing the deferred pipeline + tensor-
accelerated upscale; the CPU oracle is the measurement-methodology
stand-in. The gate is `continue-on-error: true` in CI per ADR-047 §7
— PRs do not fail on the CPU numbers.

Reading the numbers: the bench captures the methodology
end-to-end — owned arg parser, owned JSON report, p99/σ stats,
budgets-file evaluation, gate verdict — even though the workload is
order-of-magnitude different from the eventual GPU workload. PR 6
swaps the synthetic gradient for the full `combined_deferred_scene`
CPU oracle; both paths run on the dev machine. The RX 6700 XT runner
runs the GPU path and produces the gate-of-record numbers.

## How to refresh

```bash
cargo run --release -p engine-bench-frame-pacing -- \
    --run --frames 60 --output-path /tmp/phase-5-milestone.json

# Optionally evaluate against the shipped budgets:
cargo run --release -p engine-bench-frame-pacing -- \
    --gate /tmp/phase-5-milestone.json \
    --budgets tools/frame-pacing/budgets.toml
```

The bench takes ~ 1 s end-to-end on the developer machine. Append a
row to the History table on each refresh — keep the trace flat so
PR 6's GPU-baseline transition is visible.

## Transition to PR 6

PR 6 replaces the synthetic HDR gradient with the full deferred path
(`engine_raster::sample::combined_deferred_scene` per frame, followed
by bilinear upscale). The same JSON shape applies; the workload
changes. PR 6 also activates `tools/frame-pacing/budgets.toml` as the
default gate file in CI (still `continue-on-error: true` until the
runner is attached).

When the self-hosted RX 6700 XT runner is provisioned:
1. Register it as `self-hosted-gpu-rx6700xt` per the runbook
   (`docs/runbooks/frame-pacing-runner.md`).
2. Drop `continue-on-error: true` from `.github/workflows/ci.yml`.
3. Mark the `frame_pacing` job as a required status check on the
   default branch.
4. Capture the first GPU-baseline measurement here under "History".

The CPU oracle stays in the bench harness as the developer-machine
quick-check; the runner's GPU numbers are the gate of record.
