# parallel-schedule speedup baseline

Generated by `just parallel-schedule-baseline` (see ADR-033). Measures
the speedup curve of `Schedule::run_on` vs `Schedule::run` as the worker
count climbs, on a workload with multiple non-conflicting systems —
i.e. the case where `run_on` actually has parallelism to exploit.

Reuses the `tests/replay_parity.rs` workload (six systems, 1000
entities, six-system phase mix) as the harness. Captures
wall-clock-per-frame at worker counts {1, 2, 4, N} and reports the
speedup over the single-threaded reference.

The baseline is informational — not a CI gate. The Phase 3 milestone
gate is the 1 M-entity / 60 FPS target tracked in
[million-entities-baseline.md](million-entities-baseline.md); this file
records the *shape* of the speedup so a future regression in JobGraph
dispatch overhead is visible.

## How to refresh

```bash
# Until a dedicated bench harness lands, use the replay-parity oracle
# as the workload — wrap it in `time` to get a wall-clock.
for w in 1 2 4 $(nproc); do
    echo "--- workers=$w ---"
    time ENGINE_PARALLEL_WORKERS=$w cargo test \
        -p engine-core --test replay_parity --release -- --nocapture
done
```

A purpose-built criterion bench is a Phase 4+ follow-up — the Phase 3
deliverable is the deterministic dispatch correctness gate (replay
parity), not the speedup curve.
