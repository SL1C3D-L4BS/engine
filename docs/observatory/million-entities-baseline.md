# million-entities baseline

Generated by `just million-entities-baseline` (see ADR-033). Reports
per-frame wall-clock at three scales — 10k, 100k, 1M entities — for
both the sequential `Schedule::run` path and the parallel
`Schedule::run_on` path on an N-worker pool.

The Phase 3 milestone gate is **1 M entities at 60 FPS on one core**
(median frame time ≤ 16.6 ms). This baseline is informational — not a
CI gate. Runner noise makes a hard threshold infeasible in shared CI;
the milestone is verified on a developer machine and captured here.

Workload: every entity carries `Position(f32,f32,f32)` +
`Velocity(f32,f32,f32)`; one system per frame advances Position by
Velocity. Pure Table-archetype traversal across one column pair — the
hot path the archetype-SoA layout (ADR-031) was designed for.

## Bring-up note (2026-05-20, dev machine)

First-light numbers from the bench harness on the developer machine
(Linux 7.x, CachyOS, BORE scheduler). Recorded with
`cargo bench -p engine-core --bench million_entities` (release
build, default sample size 10).

| Scale | Sequential median | Parallel median |
|-------|------------------:|----------------:|
| 10k   | ~0.18 ms          | ~0.20 ms        |
| 100k  | ~1.83 ms          | ~1.95 ms        |
| 1M    | **~33 ms**        | ~33 ms          |

The 1M sequential number is **over** the 16.6 ms milestone target. The
cause is the collect-then-mutate pattern the current query DSL forces
on systems that need a `(Mut<A>, &B)` join — every frame allocates a
24 MiB `Vec<(Entity, f32, f32, f32)>` then writes it back through
`get_mut::<Position>(entity)`, which is two cache walks per row
instead of one. The Phase 4 query DSL upgrade (planned: `WorldQuery`
impls for `(Mut<A>, &B)` and `(Mut<A>, Mut<B>)`) replaces both passes
with a single archetype-stream walk and closes the gap.

The parallel path tracks the sequential one because the workload is a
single system; `run_on` only buys parallelism across multiple systems
in a phase. The million-entities benchmark exists to track
*regressions* in `Schedule::run_on` on the hot path, not to compete
with `Schedule::run` on a one-system workload.
