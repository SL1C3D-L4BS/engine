# archetype traversal baseline

Generated by `just archetype-baseline` (see ADR-031). The workload streams a
`Vec3` position component across a single archetype, then sweeps the number
of Table components per archetype to measure the impact of column count on
cache behaviour. Reruns on every dev host; commit results below alongside
the existing history.

The baseline is informational — not a CI gate. The Phase 3 milestone gate
is the 1 M-entity / 60 FPS single-core target captured in
`million-entities-baseline.md` once PR 3 lands.

## Methodology

- Build profile: `--release`.
- Workload: `cargo run --release -p cache-observatory --
  --workload archetype-traversal` (placeholder; the
  cache-observatory binary will gain the `archetype-traversal` mode once
  the PR 3 benchmark harness lands).
- Sample size: 10 k / 100 k / 1 M entities.
- Component-count sweep: 1, 4, 16 Table components per archetype, each
  16-byte aligned.
- Reported: ns / entity (median of three runs) and L1d miss rate per
  entity (kernel `perf_event` counters, ADR-027 / ADR-030).

## Notes

- The `archetype-traversal` cache-observatory mode is staged in this
  Phase 3 PR (ADR-031) as a follow-up to the existing
  `vec3_array_traversal` and `hot_cold_parallel` workloads. The numbers
  above land when PR 3's million-entities bench harness exists; until
  then the table is a placeholder so the baseline file is present and
  versioned alongside the ADR.
- The intent of the sweep is to surface the column-count cliff: more
  columns per archetype means more parallel allocations the CPU must
  prefetch per entity, and the slope of ns/entity vs column-count is the
  observable proxy for "how much does archetype-SoA help vs the foundation
  layer's flat per-component columns".
